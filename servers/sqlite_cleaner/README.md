# SQLite Cleaner MCP Server

## 概要

SQLite MCPサーバーの終了タイミング問題に対するワークアラウンドとして開発された、空機能のMCPサーバーです。

## 背景

SQLite MCPサーバーは、gemini-cli終了時に他のMCPサーバー（fetch、filesystem、puppeteer等）と異なり、コンテナが残存する問題があります。この問題はSQLiteサーバー固有のプロセス終了処理に起因しており、複数の解決策を検討した結果、最も確実で非侵襲的なアプローチとして本サーバーを実装しました。

## 検討した他のアプローチ

以下の手法を試行しましたが、それぞれに課題がありました：

### Dockerレベルでの対処
- **`--init` フラグ**: PID 1 シグナル処理改善 → 効果なし
- **`--stop-timeout=5`**: 強制終了時間短縮 → 根本解決に至らず

### プロセス監視によるアプローチ  
- **Dockerfile修正（バックグラウンド実行 + 監視）**: gemini-cli起動不可
- **ファイル監視（STDIN横取り）**: MCPプロトコル通信が破綻

### 外部スクリプトによる対処
- **独立したクリーンアップデーモン**: 運用が複雑化
- **手動エイリアス**: 毎回の手動実行が必要

### 採用した理由
上記の課題を踏まえ、以下の利点を持つ現在のアプローチを採用：
- MCPプロトコル内で完結
- 既存システムへの影響なし
- 確実な終了タイミング保証
- 運用上の追加作業なし

## 役割

- **機能**: 空のMCPサーバー（tools、resources、prompts なし）
- **目的**: gemini-cli終了時の確実な終了を利用したクリーンアップ処理
- **対象**: 残存するSQLiteコンテナの自動停止

## 動作原理

1. gemini-cli起動時、SQLiteサーバーと共に本サーバーも起動
2. gemini-cli終了時、本サーバーが確実に終了
3. 終了時に`atexit`フックが発動し、残存SQLiteコンテナを検出・停止
4. システム全体のクリーンアップが完了

## 技術仕様

- **フレームワーク**: FastMCP (公式MCP Python SDK)
- **終了処理**: Python `atexit` モジュール
- **コンテナ制御**: Docker CLI (Docker socket マウント)
- **対象**: `mcp_sqlite:latest` イメージのコンテナ

## 実装の利点

- **非侵襲的**: 既存のSQLiteサーバーに変更なし
- **確実性**: MCPライフサイクルに依存した確実な実行
- **汎用性**: 他のMCPクライアント（Claude Desktop等）でも有効
- **保守性**: シンプルな実装で理解・修正が容易
